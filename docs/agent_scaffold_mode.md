# Agent Scaffold Mode

`agent-scaffold` mode consumes an existing agent manifest and generates a minimal runnable scaffold project.

It is separate from `agent-manifest`:

- `agent-manifest`: turns requirements into `manifest.json`.
- `agent-scaffold`: turns `manifest.json` into scaffold files, tests, and a safe dry-run runner.
- `agent-project`: runs the end-to-end requirement -> manifest -> scaffold -> dry-run -> scaffold tests pipeline.

It is not the default CTCP project-generation flow. Ordinary CTCP project generation still uses `new-run`, `status`, and `advance`.

## Entrypoint

```powershell
.\.venv\Scripts\python.exe scripts\ctcp_orchestrate.py `
  agent-scaffold `
  --manifest tests\agent_factory_benchmark\generated\output_devops_incident.json `
  --output-dir runs\agent_scaffold_devops
```

Independent scaffold generator entrypoint:

```powershell
.\.venv\Scripts\python.exe scripts\generate_agent_scaffold.py `
  --manifest tests\agent_factory_benchmark\generated\output_devops_incident.json `
  --output-dir runs\agent_scaffold_devops
```

Use `--force` only to replace a directory that was previously generated by `agent-scaffold`. The consumer refuses to overwrite a non-scaffold directory.

## Scaffold Shape

The generated scaffold contains:

```text
agent_project/
  manifest.json
  README.md
  agents/
  tools/
  workflows/workflow.json
  memory/memory_schema.json
  permissions/permissions.json
  guardrails/guardrails.json
  tests/
    test_manifest_contract.py
    test_permissions.py
    test_workflows.py
    test_dry_run.py
  audit/dry_run_audit.jsonl
  sample_input.json
  run_agent.py
```

The original manifest is copied to `manifest.json` and remains the source of truth. Agent and tool definitions are split into separate JSON files so they can be reviewed independently.

## Dry Run

The scaffold is not a full LLM agent runtime. `run_agent.py` supports only dry-run inspection:

```powershell
python run_agent.py --dry-run --input sample_input.json
```

Dry-run output includes:

- selected agent
- workflow start state
- low-risk tools available for dry-run
- approval-required actions
- high-risk tools listed as pending approval
- active guardrails
- audit event and `audit/dry_run_audit.jsonl`

High-risk tools are not executed. Tools with `requires_approval=true` are reported as pending approval only.

## Scaffold Tests

Run generated scaffold tests from the scaffold root:

```powershell
python -m unittest discover tests -v
```

The tests validate manifest structure, high-risk approval requirements, audit-log requirements, and workflow flow fields.

## End-to-End Mode

Use `agent-project` when starting from a requirement input rather than an existing manifest:

```powershell
.\.venv\Scripts\python.exe scripts\ctcp_orchestrate.py `
  agent-project `
  --input tests\agent_factory_benchmark\fixtures\input_devops_incident.json `
  --output-dir runs\agent_project_devops
```

See `docs/agent_project_pipeline.md` for the full pipeline output and report contract.

## Current Limits

- The scaffold is a contract/testing harness, not a production agent runtime.
- It does not call LLMs or external tools.
- It preserves manifest permissions but does not implement human approval UI.
- It is intended for review, dry-run, and regression testing before a real runtime is implemented.
